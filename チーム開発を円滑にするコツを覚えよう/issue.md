# 課題 1（質問）

## PR を小さい粒度で作成するべき理由

適切な小さなサイズとは 1 つの自己完結型の変更であるべき

### レビューが容易になる

- 小さい PR は変更の範囲が限定的であるため、レビュワーが理解するのが容易
- 大きな PR は時間がかかるだけでなく、エラーやバグを見逃しやすくなる可能性がある

### マージの容易さ

- 小さな変更はコンフリクト（競合）が起きにくいため、マージするのが容易
- 大きな PR では、レビューとマージの間に他の変更が加えられ、新たなコンフリクトが発生する可能性がある

### リスクの軽減

- 小さな PR は、問題が発生した場合でも影響範囲が小さいため、修正も容易
- 大きな PR では一度に多くの変更を適用するため、問題が発生したときのリスクが大きくなる
- ロールバックもしやすくなる

### 効率的な開発フロー

- 小さい PR を頻繁にマージすることで、新たな機能や修正を早く反映させることができる
- 継続的なインテグレーション（CI）と継続的なデリバリー（CD）を実現できる

## PR を大きい粒度で作成せざるを得ない場面

- 例えば大規模なリファクタリングや機能追加など、一連の変更が密接に関連していて分割が困難な場合
- しかし、これらの場合でも可能な限り小さな単位に分割し、逐次レビューとマージを行うことが推奨される

## コードのコメントに書くべきこと、 書くべきではないこと

- 開発者はわかりやすい言葉でコードの目的、使用方法、使用時の動作を書くべき
- コメントは、コードが存在する理由を説明する場合に役立つが、コードが何を実行しているかを説明するべきではない
- コード自体が説明できるほど明確ではない場合は、コードをより単純にする必要がある
- ただし、いくつかの例外はある
  - たとえば、正規表現や複雑なアルゴリズムでは、何を行っているかを説明するコメントから大きなメリットが得られることがよくある
- ほとんどのコメントは、決定の背後にある理由など、コード自体には含まれない可能性がある情報を提供するためのもの

## Git などのコミットコメントに書くべきこと、書くべきではないこと

### コミットメントに書くべきこと

#### 変更の要約

- コミットの主要な変更点を短く要約したメッセージを書くことが最も重要
- これは、他の開発者が変更の概要をすばやく理解するのに役立つ

#### 変更の理由

- 特定の変更が行われた理由を記述する
- これにより、他の開発者がコードの意図を理解し、後で問題が発生したときにその原因を探るのに役立つ

#### 影響範囲

- 変更が他のコードや機能にどのような影響を及ぼすかを説明する

### Git などのコミットコメントに書くべきではないこと

#### 詳細なコードの変更

- コミットメッセージは要約であるべきで、具体的なコードの変更については、差分を見て理解すべき

#### 冗長な情報

- コミットメッセージは短く、一目で理解できるようにするべき。不必要な情報や重複した情報は避ける

#### 不明確な言葉やジャーゴン

- コミットメッセージは他の開発者にとってわかりやすくあるべきで、具体的かつ明瞭な言葉を使用するべき
